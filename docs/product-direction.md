# Product Direction

## 1. 제품 정의

`AI Leadership Lab for Team Leaders`는 다양한 산업의 팀장들이 AI를 활용해 리더십 상황을 구조화하고, 팀 실행 기준을 만들며, 구성원과의 1on1 대화까지 실천할 수 있도록 설계된 범용 리더십 Journey 웹앱입니다.

기존 `종근당/C1바이오 제약영업팀장 AI 리더십 Lab Journey v41`은 제약영업팀장 전용으로 안정화된 별도 제품입니다. 이 프로젝트는 기존 v41을 보존하면서, 그 구조적 장점만 가져와 범용 팀장용 제품으로 재구성합니다.

## 2. 핵심 사용자

1차 MVP의 핵심 사용자는 특정 산업에 한정되지 않는 `현업 팀장`입니다.

- 제조업 생산·품질·기술 팀장
- 공공기관 부서장·파트장
- 금융/보험 영업·지원 조직 팀장
- IT/프로젝트 조직의 PM·파트장
- 서비스센터 운영 팀장
- 교육, 컨설팅, 지원부서 등 지식서비스 조직 팀장

## 3. 제품 목표

- 팀장이 현재 팀 상황을 더 선명하게 볼 수 있도록 돕는다.
- 팀의 역할, 업무 기준, 실행 우선순위, 병목을 정리하게 한다.
- AI를 단순 답변 도구가 아니라 사고 확장 파트너로 활용하게 한다.
- 최종적으로 구성원 1on1 대화와 현업 실행으로 연결한다.
- 산업별 버전으로 확장할 수 있는 공통 Journey 구조를 만든다.

## 4. 유지할 v41 핵심 구조

기존 v41에서 유지할 핵심은 `10단계 Journey 구조`, `단계별 입력-정리-실행 흐름`, `AI 과제형 학습 경험`, `팀장 자기 적용 중심`입니다.

| 단계 | 범용 팀장용 단계명 | 핵심 의도 |
| --- | --- | --- |
| 1 | 시작하기 | 현재 리더십 과제와 사용 목적을 확인한다. |
| 2 | 팀과 역할 보기 | 팀의 역할, 구성, 책임 범위를 파악한다. |
| 3 | 리더 질문 다듬기 | 막연한 고민을 좋은 질문으로 바꾼다. |
| 4 | 산업·고객·조직 변화 읽기 | 외부 변화와 내부 요구를 함께 해석한다. |
| 5 | 우리 팀 실행 기준 만들기 | 팀이 지켜야 할 업무 기준을 정한다. |
| 6 | 핵심 업무 실행계획 만들기 | 중요한 업무를 실행 가능한 계획으로 바꾼다. |
| 7 | 할 일·줄일 일 정하기 | 집중할 일과 덜어낼 일을 구분한다. |
| 8 | 업무 경계와 병목 대응 | 역할 경계, 협업 충돌, 병목을 정리한다. |
| 9 | 1on1 대상 선택 | 우선 대화가 필요한 구성원을 선택한다. |
| 10 | 1on1 대화 실천 | 실제 대화 스크립트와 실행 약속으로 마무리한다. |

## 5. 제거할 제약영업 특화 표현

다음 표현은 범용 팀장용 제품에서 제거하거나 치환합니다.

- 제약영업팀장
- 종근당
- C1바이오
- 병원, 의원, 거래처
- 의사, 원장, 약사 등 제약영업 고객 호칭
- 처방, 처방 확대, 처방 전환
- 제품명, 포트폴리오, 품목
- 방문, 영업활동, 콜 활동
- 고객 반응, 고객 방문 기록
- Follow-up, 리마인드 콜
- 지점, 영업소, MR 등 특정 영업조직 용어

## 6. 범용 팀장용 표현 체계

| 기존 제약영업 표현 | 범용 표현 | 사용 맥락 |
| --- | --- | --- |
| 제약영업팀장 | 팀장, 현업 팀장 | 사용자 호칭 |
| 영업팀 | 우리 팀, 현업 팀 | 조직 단위 |
| 병원/의원 | 고객, 이해관계자, 현장 | 외부·내부 고객 |
| 처방/제품 | 성과 과제, 서비스, 프로젝트, 핵심 업무 | 성과 대상 |
| 영업활동 | 실행 활동, 업무 수행, 현장 대응 | 행동 단위 |
| 고객 반응 | 현장 신호, 이해관계자 반응, 구성원 반응 | 상황 해석 |
| 다음 방문 | 다음 행동, 후속 조치 | 실행 연결 |
| 팀장 코칭 필요 | 리더 개입 필요, 1on1 필요 | 리더 행동 |

## 7. 제품 톤앤매너

- 교과서적 설명보다 `팀장이 바로 입력하고 적용할 수 있는 문장`을 우선한다.
- 특정 산업을 모르는 사용자도 이해할 수 있는 보편적 업무 언어를 사용한다.
- 지나치게 추상적인 리더십 용어보다 `업무 기준`, `다음 행동`, `병목`, `역할 경계`, `대화` 같은 실행 언어를 사용한다.
- AI 결과는 완성 답안이 아니라 팀장의 판단을 돕는 초안으로 안내한다.

## 8. 산업별 확장 전략

1차 MVP는 공통 Journey와 범용 콘텐츠로 개발합니다. 이후 산업별 확장은 config 기반을 우선 검토합니다.

### 8.1 공통 Core

- 10단계 Journey 구조
- 단계 전환 로직
- 입력/출력 패턴
- 저장 또는 복사 기능
- AI 과제 지시문 템플릿
- 공통 QA 기준

### 8.2 산업별 Config 후보

- 산업명
- 사용자 역할명
- 대표 고객/이해관계자
- 대표 업무 과제
- 대표 병목 상황
- 1on1 대화 유형
- 예시 문장
- 금지/주의 표현

### 8.3 산업별 Route 후보

```text
/journey-teamleader-preview.html
/journey-manufacturing-preview.html
/journey-public-sector-preview.html
/journey-finance-sales-preview.html
/journey-it-project-preview.html
/journey-service-center-preview.html
```

## 9. 새 route 전략

초기 기본 route는 `/journey-teamleader-preview.html`입니다.

운영 원칙은 다음과 같습니다.

- 기존 v41 route `/journey-v41-preview.html`는 참조만 하고 수정하지 않는다.
- 새 프로젝트의 preview route는 범용 팀장용으로 독립 운영한다.
- 산업별 route는 동일한 core를 공유하고 config만 바꾸는 방향을 우선 검토한다.
- route가 늘어나더라도 10단계 Journey의 학습 흐름은 일관되게 유지한다.

## 10. GitHub/Vercel 개발 원칙

- `main`은 안정 기준 브랜치로 유지한다.
- 기능 추가, 문구 치환, route 추가는 작은 브랜치 단위로 진행한다.
- 가능하면 모든 변경은 PR로 검토한다.
- 기존 안정화 저장소 `ckd-ci-bio-decision-v1`은 수정하지 않는다.
- Vercel 프로젝트는 기존 v41 프로젝트와 분리한다.
- preview 배포에서 route 동작을 확인한 뒤 병합한다.
- PR 본문에는 변경 목적, 변경 파일, QA 결과, 남은 과제를 명시한다.

## 11. QA 기준

### 11.1 콘텐츠 QA

- 제약영업 특화 표현이 남아 있지 않은가?
- 범용 팀장에게 자연스러운 표현인가?
- 산업별 확장 시 config로 분리 가능한 표현인가?
- 팀장의 실제 실행 행동으로 연결되는가?

### 11.2 기능 QA

- 10단계가 순서대로 이동하는가?
- 입력값이 단계별로 자연스럽게 이어지는가?
- 복사, 저장, 출력 등 기존 핵심 기능이 유지되는가?
- 새로고침 또는 이동 시 치명적 오류가 없는가?

### 11.3 배포 QA

- `/journey-teamleader-preview.html`이 정상 로드되는가?
- Vercel preview URL에서 CSS/JS 경로가 깨지지 않는가?
- 기존 v41 URL과 저장소에 영향이 없는가?

## 12. 1차 MVP 범위

1차 MVP는 `범용 팀장용 10단계 Journey`를 만드는 데 집중합니다.

포함 범위:

- 기본 route 구성
- 10단계 화면 구조
- 범용 문구 치환
- 팀장용 입력 예시
- AI 과제 지시문 초안
- 1on1 대화 실천 흐름
- 최소 QA 체크리스트

제외 범위:

- 산업별 상세 버전 전체 제작
- 로그인/계정 기능
- 서버 저장소
- 복잡한 관리자 페이지
- 새로운 대규모 디자인 개편
- 기존 v41 기능 변경
